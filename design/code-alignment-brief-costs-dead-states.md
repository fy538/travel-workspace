## Task C-1 — `disputed` ledger-row state is dead code [BACKEND-GATED — flag, do not fake]
`utils/costsViewModel.ts:14-21` defines `CostsLedgerRowState` including `'disputed'`, and `components/expense/CostsLedgerRow.tsx:17,52-56,156-172` fully renders it (oxblood tone, `!` badge on the avatar, "Ana flagged"-style copy per canon `costs-screens.jsx:59,128`). But `buildLedgerRow` (`costsViewModel.ts:333-375`) never assigns `'disputed'` — its only three possible outputs are `pending | settled | youPaid/otherPaid/normal` (`costsViewModel.ts:342-352`). Checked the backend `Expense` schema (`utils/api/schema.gen.ts`, `Expense: {...}`) and `ExpenseComment` (`id/expense_id/user_id/content/created_at` only) — there is no flag/dispute field anywhere in the data model a client could read to produce this state.
- Do not synthesize "disputed" from any client-side heuristic (e.g. treating a comment as a flag).
- **Flag as backend schema work**: dispute needs an explicit field (e.g. `expense.flagged_by` / a dispute reason on `ExpenseComment`) before this state can ever render. Until then, either (a) leave the dead branch in place with a comment noting it's BE-gated (current state, acceptable), or (b) remove the case from `CostsLedgerRowState` if the team wants the type to reflect only reachable states — team's call, not a forced change.
- Action for this task: add a one-line code comment at `costsViewModel.ts:14-21` and `CostsLedgerRow.tsx:11-19` marking `disputed` as BE-gated/unreachable pending a dispute field, so a future reader doesn't assume it's wired and doesn't try to fake it from a comment or client-side flag.

## Task C-2 — `reimbursement` ledger-row state is dead code [BACKEND-GATED — flag, do not fake]
Canon (`costs-screens.jsx:63,131`, `costs-system.jsx:107-120`) shows a distinct reimbursement row ("Theo paid you back · SETTLED IN CASH") for a cash-settlement event logged after the fact. `CostsLedgerRowState` includes `'reimbursement'` and `CostsLedgerRow.tsx:31-46` renders the full swap-glyph treatment for it, but nothing in `buildLedgerRow` ever emits it. There is no reimbursement/cash-settlement transaction concept in `TripSettlement` / `SettlementTransfer` / `BalanceSummary` (checked `schema.gen.ts` — transfers are computed net-owed suggestions, not logged settlement events) and no expense `category`/`source` value represents "someone paid someone back in cash."
- Same disposition as C-1: this needs a backend concept (a logged settlement/reimbursement event, distinct from the computed `SettlementTransfer` suggestions) before the FE state is reachable.
- Action for this task: add the same BE-gated comment at the `reimbursement` branches in `costsViewModel.ts` and `CostsLedgerRow.tsx`. Do not wire a client-side "mark as paid back" affordance that fabricates a reimbursement row without a real backend-logged event — that would let two travelers each believe a debt is cleared without a shared source of truth.
